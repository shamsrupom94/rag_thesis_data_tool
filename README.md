# RAG Thesis Q&A Dataset Reviewer

This project is a local review tool for manually checking and curating the LLM-generated question-and-answer datasets created for our thesis. The datasets include academic papers about Retrieval-Augmented Generation (RAG) and NIST SP 800 publications, then reviewed by humans to make sure each question, answer, and evidence reference is correct before the data is used in the thesis workflow.

The main purpose of the project is to make manual validation easier. Instead of editing many JSON files by hand, the tool gives reviewers a structured interface where they can open each paper, inspect its extracted sections, compare the generated Q&A pairs with the original evidence, update review status, edit answers, add tags, create missing Q&A pairs, or remove incorrect entries.

## Project Purpose

The thesis datasets contain Q&A pairs generated by an LLM from academic RAG papers and NIST SP 800 publications. Because LLM-generated data can contain incorrect answers, weak evidence, incomplete citations, or unclear wording, the datasets need manual curation. This project supports that curation process by connecting three important sources:

- the original PDF files,
- the extracted and sectionized text from those documents,
- the generated Q&A JSON files that are being reviewed.

Each reviewed Q&A item can be marked as `verified`, `unverified`, or `needs_revision`. This makes the dataset easier to track and improves confidence that the final curated data is grounded in the source documents.

## Backend

The backend is a FastAPI application located in `backend/`. Its purpose is to provide a safe local API over the thesis data files in `data/` and `data_nist/`.

The backend:

- discovers available academic documents from `data/acad_rag_sect_docs/`,
- discovers available NIST SP 800 documents from `data_nist/nist_sp_sect_docs/`,
- loads the matching Q&A files from `data/qna_dataset/`,
- links documents to their original PDFs in `data/academic_pdf/`,
- serves PDFs to the frontend for side-by-side review,
- validates Q&A edits against known document sections,
- creates, updates, and deletes Q&A pairs,
- deletes a full document and its related extracted files when needed,
- writes updated Q&A JSON files atomically so local data is not partially saved.

The backend does not train or generate new LLM data. Its role is to manage the existing thesis dataset and preserve the reviewed changes made by human curators.

## Frontend

The frontend is a React + Vite application located in `src/`. Its purpose is to give reviewers a practical workspace for checking the dataset without manually navigating raw files.

The frontend allows reviewers to:

- switch between the Academic RAG and NIST SP 800 collections,
- browse all documents in the selected local collection,
- see the number of sections and Q&A pairs per paper,
- track review progress for each document,
- search and filter Q&A pairs by status,
- edit the question, exact answer, LLM-generated answer, answer type, tags, difficulty, and review status,
- select the relevant source section for a Q&A pair,
- paste or correct the evidence sentence used to support the answer,
- inspect the extracted section text and original PDF,
- add new Q&A pairs when the dataset needs more coverage,
- delete incorrect Q&A pairs or remove an entire document from the local dataset.

The frontend is designed as a local review desk for thesis data curation, not as a public web application.

## Q&A Dataset

The curated academic Q&A dataset lives in `data/qna_dataset/`, and the NIST SP 800 Q&A dataset lives in `data_nist/qna_dataset_nist/`. Each file corresponds to one source document, for example:

```text
data/qna_dataset/acad_0001_sections_qna.json
data_nist/qna_dataset_nist/nist_0001_sections_qna.json
```

Each Q&A entry includes fields such as:

- `qid`: the unique question ID,
- `doc_id` and `doc_title`: the source document,
- `question`: the generated or edited question,
- `answer_exact`: the concise evidence-grounded answer,
- `answer_llm`: the longer LLM-style answer,
- `answer_type`: the category of the question, such as facts, methods, results, definitions, comparisons, or analysis,
- `relevant_sections`: the source section, evidence sentence, and character offsets,
- `tags`: topic labels for organization,
- `difficulty`: easy, medium, or hard,
- `status`: the manual review state.

The goal of manual curation is to verify that each Q&A pair is accurate, understandable, and traceable to the source paper. The final dataset can then be used more reliably in thesis experiments, evaluation, or analysis related to RAG systems.

## Data Folders

The project expects the following data layout:

```text
data/
  academic_pdf/             Original academic PDF files
  academic_extracted/       Extracted Markdown text from the PDFs
  acad_rag_sect_docs/       Sectionized JSON files for each paper
  qna_dataset/              LLM-generated and manually curated Q&A files

data_nist/
  nist_pdf/                 Original NIST SP 800 PDF files
  nist_extracted/           Extracted Markdown text from the PDFs
  nist_sp_sect_docs/        Sectionized JSON files for each NIST document
  qna_dataset_nist/         LLM-generated and manually curated Q&A files
  nist_sectionized.jsonl    Combined sectionized NIST rows
```

By default, the backend reads academic data from `./data` and NIST data from `./data_nist`. To override those roots, set `RAG_REVIEWER_DATA_DIR` for academic data and `RAG_REVIEWER_NIST_DATA_DIR` for NIST data.

## Run Locally

Use Python 3.10+ and Node.js 18+.

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1
pip install -r backend\requirements.txt
npm install
```

Start the API and UI in separate terminals:

```powershell
uvicorn backend.main:app --reload
npm run dev
```

Open the frontend at:

```text
http://localhost:5173
```

The backend API documentation is available at:

```text
http://127.0.0.1:8000/docs
```

## Notes

- The project is intended for local thesis data review.
- Q&A changes are saved back into the selected collection's Q&A folder.
- The API validates section references before saving edits.
- Review progress is tracked through the `status` field on each Q&A pair.
